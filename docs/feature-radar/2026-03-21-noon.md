## New Ideas (Beyond Morning Radar)

### 1. "Vibe Dashboard" — Natural-Language Layout Builder Inspired by Google Stitch

**Market signal:** Google redesigned Stitch (March 19) into an AI-native infinite canvas with "Vibe Design" — users describe a business objective or desired feeling and the AI generates multiple UI directions. Voice Canvas lets users speak to the canvas while the AI asks clarifying questions and makes live updates. Stitch already integrates with Claude Code and Cursor, blurring the line between design and development.

**Feature idea:** Apply the "vibe design" paradigm to Winston's dashboard composition. Today, dashboard composition translates a natural-language *query* into widgets. A Vibe Dashboard mode would let users describe an *outcome* — "I want a board-ready portfolio overview that emphasizes risk exposure" — and Winston generates an entire dashboard layout with widget selection, ordering, color emphasis, and data bindings. Users refine by talking: "Make the debt maturity chart more prominent" or "Add a geographic heat map." This is compositional design, not just compositional querying.

**Which module:** Enhancement to `backend/app/services/dashboard_composer.py` (layout generation logic), new "Vibe Mode" toggle in `repo-b/src/app/app/dashboards/`, optional voice input via Web Speech API

**How it differs from existing radar:** Morning radar Feature #4 (Workflow Orchestration) is about chaining MCP tools sequentially. This is about *spatial* composition — generating and refining visual layouts through conversational iteration. Dashboard Composition already exists in the capability inventory, but it's query→widget, not outcome→layout.

**Enterprise value:** Board meetings and LP presentations require dashboards tailored to audience and narrative. Today, users manually arrange widgets. A vibe-driven composer lets a portfolio manager say "Build me the quarterly LP review dashboard" and get a curated layout in seconds. Reduces dashboard creation from 30 minutes to 30 seconds.

**Implementation complexity:** Medium — Dashboard Composer service exists (38KB). This extends it with a layout-generation prompt layer and a conversational refinement loop. Voice input is optional and uses browser-native APIs. No new infrastructure.

**Priority score:** 6

**Priority rationale:** Builds on existing infrastructure with moderate effort. Differentiating UX that no REPE competitor offers. Not urgent like compliance (radar Feature #1) but high demo value and directly showcases Winston's AI-native advantage over spreadsheet-based competitors.

### 2. Autonomous Document Room Agent — Inspired by GPT-5.4 Document Manipulation

**Market signal:** OpenAI's GPT-5.4 (March 5) introduced agents that can autonomously browse websites, fill forms, and manipulate documents. This shifts document processing from "upload and extract" to "go find, pull, and process." Separately, Perplexity launched domain-specific agent hubs (Perplexity Health) with specialized sub-agents — validating the pattern of purpose-built agents within a platform.

**Feature idea:** Build a "Data Room Agent" that can autonomously monitor and process documents from external data rooms (Intralinks, Datasite, ShareFile). When a new document appears in a deal's data room, the agent: (a) detects the document type (rent roll, T-12, appraisal, environmental report), (b) runs the appropriate extraction profile from Winston's existing extraction engine, (c) populates the deal record with extracted metrics, and (d) flags anomalies or missing documents against a configurable checklist. The agent operates on a polling schedule with a human-reviewable log of all actions taken.

**Which module:** New `backend/app/services/data_room_agent.py`, integration with existing `backend/app/services/extraction_engine.py` and `backend/app/services/documents.py`, new MCP tools for data room polling, new UI in `repo-b/src/app/app/deals/[dealId]/data-room/`

**How it differs from existing radar:** Morning radar Feature #2 (Data Integration Hub) is about pulling structured data from Preqin/PitchBook. This is about *unstructured document processing* from deal-specific data rooms — a fundamentally different data source and workflow. Morning radar Feature #4 (Workflow Orchestration) is generic; this is a purpose-built agent for a specific high-value REPE workflow.

**Enterprise value:** Deal teams spend 5-10 hours per deal manually downloading, categorizing, and extracting data from data rooms during due diligence. An autonomous agent that handles the mechanical work lets analysts focus on judgment calls. At firms evaluating 50+ deals/year, this saves hundreds of analyst hours. Directly addresses a pain point that Dealpath and Juniper Square have not solved with AI.

**Implementation complexity:** Medium-High — Winston already has document extraction (extraction engine + profiles) and document management. The new work is the data room polling layer (API integrations with Intralinks/Datasite), the document type classifier, and the autonomous loop with logging. The extraction pipeline is reused.

**Priority score:** 7

**Priority rationale:** High enterprise value on a pain point no competitor has solved with AI. Builds on Winston's existing extraction engine rather than starting from scratch. The data room API integrations are the main new dependency. Good candidate for a phased rollout: Phase 1 = manual upload + auto-classify + auto-extract, Phase 2 = polling integration.

### 3. Ultra-Cheap Continuous Portfolio Monitoring — Inspired by Gemini 3.1 Flash-Lite Pricing

**Market signal:** Google launched Gemini 3.1 Flash-Lite at $0.25 per million input tokens — a 10-40x cost reduction vs. frontier models. This pricing makes *continuous* AI monitoring economically viable, where previously AI was invoked only on user request. The trend toward sub-$1/M-token models enables always-on background intelligence.

**Feature idea:** Build a "Winston Sentinel" background monitoring layer that continuously evaluates portfolio health using ultra-cheap inference. Every 6 hours, Sentinel runs a lightweight analysis across all active assets: checking for covenant threshold proximity, debt maturity cliff warnings, occupancy trend inflection points, and variance drift from underwriting assumptions. Results surface as a prioritized "Attention Queue" in the Winston dashboard — not a notification flood, but a ranked list of "things that changed since yesterday" with one-click drill-down. Use Flash-Lite-class models for the monitoring sweeps; escalate to Opus/Sonnet only when an anomaly is detected and needs deeper analysis.

**Which module:** New `backend/app/services/sentinel.py` (monitoring loop), enhancement to `backend/app/services/ai_gateway.py` (model tiering for background tasks), new "Attention Queue" widget in `repo-b/src/app/app/dashboard/`, integration with existing covenant, debt, and portfolio analytics modules

**How it differs from existing radar:** Morning radar Feature #8 (Fine-Tuned Model) discusses cost reduction via fine-tuning — a high-complexity, long-term initiative. Sentinel uses *existing* cheap models (no fine-tuning required) for a specific use case: continuous monitoring. It's a product feature, not a model strategy. The model router mentioned in Feature #8 is one component; Sentinel is the full product built on top of cheap inference.

**Enterprise value:** REPE portfolio managers currently review asset performance reactively — during quarterly reviews or when a problem surfaces. Continuous monitoring catches covenant breaches, occupancy drops, and variance drift weeks earlier. At $0.25/M tokens, monitoring a 50-asset portfolio costs pennies per day. This transforms Winston from "tool you query" to "system that watches your portfolio."

**Implementation complexity:** Medium — All the underlying analytics (covenant tracking, debt surveillance, variance analysis) already exist in Winston's capability inventory. Sentinel orchestrates existing calculations on a schedule and ranks the outputs. The new work is the scheduling loop, the attention-ranking algorithm, and the dashboard widget. No new data sources required.

**Priority score:** 7

**Priority rationale:** Transforms Winston's value proposition from reactive to proactive — a major positioning upgrade. Low marginal cost because it reuses existing analytics modules. The "always watching" narrative is compelling for enterprise sales. Medium complexity because the analytical building blocks already exist. Pairs well with the Accuracy Scorecard (radar Feature #5) to show confidence levels on alerts.
