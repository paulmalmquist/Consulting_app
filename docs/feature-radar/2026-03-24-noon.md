### 1. ENHANCEMENT — Winston Adaptive Thinking Tiers: Upgrade Previous Radar Idea with New API

**Market signal:** Yesterday's noon scan (2026-03-23) identified "Adaptive Thinking Budget" as a MEDIUM-HIGH priority idea and suggested building a custom query complexity classifier. However, Anthropic's Opus 4.6 **already ships** four native thinking effort controls — `low`, `medium`, `high` (default), and `max` — via the Thinking API parameter. Winston does not need to build a custom classifier; it needs to wire these native tiers into the existing gateway's model dispatch layer.

**Per CAPABILITY_INVENTORY.md, Winston already has:** AI gateway (189KB service) with model dispatch, token budget management, and assistant scope. The gateway already dispatches model calls — this is a configuration upgrade, not a new architecture.

**The gap → upgrade:** The 2026-03-23 noon scan suggested building a bespoke complexity classifier. That's now unnecessary overhead. The correct implementation is: (1) map Winston's existing query intent categories (lookup vs. analytical vs. multi-step modeling) to Opus 4.6's native `low/medium/high/max` effort tiers, (2) pass the `thinking` parameter in model dispatch calls, (3) expose a `reasoning_depth` control in the Winston copilot Context Panel so power users can override. This is lower complexity than yesterday's estimate suggested.

**Enhancement scope:** Wire `thinking: {type: "adaptive", budget_tokens: N}` into the AI gateway dispatch layer. Map REPE query types: simple lookup → `low`, analytical query → `medium`, multi-step financial modeling → `high`, scenario/Monte Carlo → `max`. Expose override in UI.

### 3. ENHANCEMENT — Context Compaction for Winston Long-Running Analysis Sessions

**Market signal:** Claude Opus 4.6 ships a "Compaction API" that automatically summarizes context when approaching token limits in long-running agent sessions. For Winston, long analysis sessions (multi-turn deal due diligence, multi-step scenario modeling, extended quarterly close workflows) are exactly the use case where context rot is currently a real problem.

**Per CAPABILITY_INVENTORY.md, Winston already has:** AI gateway with token budget management, RAG system, assistant scope (context isolation per workspace), and Chat UI with History Drawer. The gateway's 189KB size suggests there's already token management logic — but no "Compaction" capability is documented.

**The gap:** Without compaction, Winston's long analysis sessions either: (a) hit the context ceiling and lose early-session context (deal documents, prior model outputs), or (b) get cut short by the token budget manager. The Compaction API lets the gateway automatically compress older context into a dense summary, freeing tokens for new reasoning without losing session history. For REPE use cases — "I've been working through this deal for 45 minutes and now I want to stress-test it" — this is a meaningful reliability improvement.

**Enhancement scope:** Integrate the Compaction API into Winston's AI gateway session manager. Trigger compaction when the conversation approaches 70% of the context window (configurable threshold). Log compaction events to the History Drawer with a "Session context compressed — [N] messages summarized" indicator so users understand why the conversation is being managed. Pair with the existing 1M-token "Deal Room Mode" idea (yesterday's radar) as an alternative: Deal Room Mode avoids chunking entirely; Compaction Mode handles long agentic sessions gracefully.

**Files likely involved:** `backend/app/services/ai_gateway.py`, `backend/app/services/ai_conversations.py`, `repo-b/src/components/copilot/` (History Drawer)

| Dimension | Score (1-5) |
|---|---|
| Signal strength | 4 — API just shipped; direct reliability improvement for Winston's real use cases |
| Complexity | 2 — Gateway already manages tokens; needs Compaction API call + UI indicator |
| Differentiation | 3 — Operational reliability differentiator; not a headline sales feature but reduces churn from bad long-session UX |

**Priority score: MEDIUM-HIGH**

### 4. ENHANCEMENT — Agent Teams Architecture: Winston Multi-Agent REPE Workflows

**Market signal:** Claude Opus 4.6 "agent teams" allow a master agent to split large tasks into segmented jobs dispatched to specialized sub-agents. Anthropic's TechCrunch coverage (February 5, 2026) specifically highlighted this as the primary architectural upgrade in Opus 4.6. The Finance Agent benchmark win is partly attributed to agent team coordination on multi-step financial tasks.

**Per CAPABILITY_INVENTORY.md, Winston already has:** 31 MCP tool categories covering every REPE domain, AI gateway with model dispatch, PDS service (3,083 lines), capital projects, financial modeling, credit decisioning, and the full copilot workspace. The platform is already a multi-surface product — it just doesn't yet coordinate those surfaces as an "agent team."

**The gap:** Winston currently processes each query in a single-agent dispatch cycle. A REPE workflow like "prepare a quarterly investor report" requires: (1) pulling fund-level financials, (2) running asset-level variance analysis, (3) generating waterfall distributions, (4) drafting LP communications. Today, these are sequential copilot interactions. With Opus 4.6 agent teams, Winston could dispatch a coordinator agent that fans out specialized sub-agents (finance agent, narrative agent, LP comms agent) in parallel and assembles the result. This directly maps to REPE "quarter close" workflows already in the product.

**Enhancement scope:** Design a "Winston Workflow Agent" mode in the AI gateway that accepts a high-level task (e.g., "Generate Q4 LP Report for Fund X"), decomposes it using a coordinator agent, fans out to 2-4 specialized sub-agents using existing MCP tool categories, and assembles outputs into an assistant block response. Start with the quarterly close workflow as the pilot. This is the highest-complexity idea in this scan — build as a phased addition to the agentic build surface (`skills/winston-agentic-build/SKILL.md`).

**Files likely involved:** `backend/app/services/ai_gateway.py`, `backend/app/mcp/` (tool schemas for sub-agent contexts), `skills/winston-agentic-build/SKILL.md` (reference), quarter-close and LP reporting services

| Signal strength | 5 — Native Opus 4.6 feature; Finance Agent benchmark win attributed to agent teams |
| Complexity | 5 — New architectural mode; requires coordinator + sub-agent design |
| Differentiation | 5 — No REPE platform currently ships multi-agent workflow automation |

**Priority score: HIGH (phased) — highest differentiation in this scan; scope carefully before building**
