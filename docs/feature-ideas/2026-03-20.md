# Feature Ideas — 2026-03-20

## 🏆 Top Idea of the Day

**MCP Apps: Interactive Widgets Inside Winston Chat Responses**

MCP Apps just went live as the first official MCP extension — tools can now return interactive UI components that render directly inside the conversation. Claude, ChatGPT, VS Code, and Goose have all shipped client support. For Winston, this is transformative: instead of a static waterfall table, `finance.run_waterfall` could return an interactive slider letting the LP adjust the promote hurdle and see distributions recalculate in real time, without a new prompt. Instead of a flat cap rate comparison, `finance.stress_cap_rate` could return a draggable sensitivity heatmap. This moves Winston from "AI that answers questions" to "AI that hands you live instruments" — a leap no REPE competitor is making. The timing is perfect: Winston already has 83 MCP tools and a streaming chat workspace; MCP Apps gives those tools a rendering surface they've never had. Implementation touches `backend/app/mcp/server.py` (return UI component schemas from tools) and the frontend StepRenderer/ToolActivityBlock to render them. Effort is ~3-5 days for the first 3 interactive tools. This is the kind of demo moment that wins enterprise POCs.

## 💡 All Ideas Today

### 1. MCP Apps Interactive Widgets in Chat — Impact: High | Effort: Med

**Source:** [MCP Apps blog post](http://blog.modelcontextprotocol.io/posts/2026-01-26-mcp-apps/) — first official MCP extension; tools return interactive UI components rendered in-conversation. Claude, ChatGPT, VS Code, and Goose now support it.

**Winston surface:** `backend/app/mcp/server.py` (tool return schemas), `repo-b/src/components/winston/blocks/` (new `InteractiveBlock.tsx` renderer), `repo-b/src/lib/commandbar/assistantApi.ts` (SSE event handling).

**Implementation sketch:**
- Upgrade the MCP server SDK in `backend/` to the version supporting the Apps extension (check `modelcontextprotocol` PyPI for latest)
- Pick 3 high-value tools for the pilot: `finance.run_waterfall` (slider for hurdle rate), `finance.stress_cap_rate` (draggable sensitivity matrix), `repe.list_assets` (filterable/sortable table)
- Each tool's return payload gains a `ui_component` field alongside the existing `content` — the component schema defines inputs (sliders, dropdowns) and a rendering template
- On the frontend, create `InteractiveBlock.tsx` that reads the component schema and renders shadcn/ui controls (Slider, Select, DataTable) bound to the component's state
- When the user changes an input (e.g., moves the hurdle slider from 8% to 10%), the component fires a callback that re-invokes the MCP tool with the new parameter — no new chat message needed
- Fall back to static rendering for clients that don't support MCP Apps (Winston's existing response_block rendering)
- Estimated time: 3–5 days for pilot (3 tools), then incremental rollout

**Competitive angle:** Juniper Square's JunieAI uses task-specific agents but delivers static text/table results. Interactive widgets inside the conversation are a fundamentally different UX — Winston becomes a live instrument, not just an answer engine.

### 2. FastAPI Pydantic-in-Rust JSON Serialization — 2x API Response Speed — Impact: Med | Effort: Low

**Source:** [FastAPI 0.135.1 release](https://fastapi.tiangolo.com/release-notes/) (March 1, 2026) — JSON responses are now serialized via Pydantic's Rust-backed engine when a Pydantic return type or response_model is present, yielding 2x+ performance improvement on JSON serialization.

**Winston surface:** `backend/` — every FastAPI route that returns a Pydantic model (most of them).

**Implementation sketch:**
- Upgrade FastAPI in `backend/requirements.txt` from current version to `>=0.135.1`
- Upgrade Starlette to `>=0.46.0` (new requirement)
- Audit routes for `response_model=` declarations — routes that already have them get the 2x speedup automatically
- Add `response_model` to routes that return dicts but have a corresponding Pydantic schema (common pattern in `backend/app/routes/` — many return `dict` instead of the schema)
- Enable the new Content-Type validation (requests without `application/json` header now rejected by default) — verify Winston's frontend sends the header on all `bosFetch()` calls
- Run `make test-backend` to catch any serialization differences (Pydantic-Rust may handle edge cases like Decimal/UUID differently)
- Estimated time: 2–4 hours

**Why now:** This is a free performance win on a dependency bump. Winston's Fund Metrics and LP Summary endpoints return large Pydantic models — 2x serialization speed directly reduces TTFB on the heaviest pages. Also picks up native SSE support (see Idea #5).

### 3. GPT-5.4 nano for Fast-Path Intent Dispatch — Impact: High | Effort: Med

**Source:** [OpenAI GPT-5.4 mini and nano release](https://releasebot.io/updates/openai) (March 2026) — smallest GPT-5.4 variants; nano outperforms GPT-5 mini on coding and reasoning while running 2x+ faster at a fraction of the cost.

**Winston surface:** `backend/app/services/ai_gateway.py` (model dispatch), `backend/app/services/repe_intent.py` (intent classification).

**Implementation sketch:**
- Currently Winston routes all queries through Claude Opus 4.6 via the AI gateway — even simple lookups like "what's the occupancy at Cascade?" that don't need 1M-context reasoning
- Add a two-tier model dispatch in `ai_gateway.py`:
  - **Tier 1 (GPT-5.4 nano):** Intent classification, entity extraction, simple metric lookups, formatting responses. ~5ms latency, minimal cost.
  - **Tier 2 (Claude Opus 4.6):** Complex analysis, multi-step reasoning, waterfall calculations, cross-document synthesis, anything requiring extended thinking.
- The intent classifier in `repe_intent.py` already categorizes queries — extend it to output a `model_tier` field
- Add OpenAI SDK to `backend/requirements.txt`, configure `OPENAI_API_KEY` env var
- Route Tier 1 queries through a lightweight OpenAI call; route Tier 2 through Claude as today
- Implement a cost tracker: log model, tokens, latency, and cost per query to enable ROI measurement
- Estimated time: 2–3 days

**Why this matters:** Winston's API costs scale linearly with user adoption. A two-tier dispatch could reduce average cost-per-query by 40-60% on typical REPE workloads (which are ~60% simple lookups) while making simple queries feel instant. This is the Cursor Composer 2 lesson applied: use the right model for the right task.

### 4. Recharts 3.8 Coordinate Hooks for Click-to-Query Charts — Impact: Med | Effort: Low

**Source:** [Recharts 3.8.0](https://github.com/recharts/recharts/releases) (released ~March 7, 2026) — new coordinate system hooks for converting between data coordinates and pixel positions, plus z-index support for chart element layering.

**Winston surface:** `repo-b/src/components/charts/` (all chart components), `repo-b/src/components/winston/blocks/ChartBlock.tsx` (response block charts).

**Implementation sketch:**
- Upgrade `recharts` in `repo-b/package.json` from current to `^3.8.0`
- Use the new `useCoordinateSystem()` hook to enable click-to-query: when a user clicks a data point on a NOI trend chart, Winston auto-generates and runs the query "Show me the GL detail behind the $2.4M NOI in Q3 2025"
- Implementation pattern:
  ```tsx
  const { dataToPixel, pixelToData } = useCoordinateSystem();
  // On click → resolve data point → inject query into chat
  ```
- Use z-index support to layer annotations (target vs. actual bands, threshold lines) on top of data series — currently Winston chart annotations render behind data bars
- Apply TypeScript generics (`data` and `dataKey` props now support generics) to get compile-time type safety on chart data — eliminates a class of runtime errors when data shape changes
- Estimated time: 4–6 hours

**UX value:** "Click a data point, get the story behind it" is the kind of interaction that makes fund managers fall in love with a tool. It turns static charts into query launchers — every visualization becomes an entry point to deeper analysis.

### 5. FastAPI Native SSE for Simplified Streaming — Impact: Med | Effort: Low

**Source:** [FastAPI 0.135.1](https://fastapi.tiangolo.com/release-notes/) — added native Server Sent Events (SSE) support and streaming for JSON Lines and binary data via `yield`.

**Winston surface:** `backend/app/routes/ai_gateway.py` (chat streaming endpoint), `backend/app/services/ai_gateway.py` (SSE event emission).

**Implementation sketch:**
- Winston currently uses a custom SSE implementation over `StreamingResponse` with manual event formatting (`data: {...}\n\n`)
- FastAPI 0.135.1 provides first-class SSE support — replace the manual SSE formatting with FastAPI's native `EventSourceResponse` or equivalent
- Benefits:
  - Automatic event ID tracking (enables client reconnection on dropped connections — currently Winston has no reconnection logic)
  - Proper `retry:` field support (tells the browser how long to wait before reconnecting)
  - Content-Type negotiation handled by the framework
  - Cleaner code: ~50 lines of manual SSE boilerplate in `ai_gateway.py` replaced by framework primitives
- Wire in alongside the Pydantic-Rust upgrade (Idea #2) since it's the same dependency bump
- Test with the existing 12 SSE event types (token, tool_call, tool_result, response_block, structured_result, citation, done, etc.)
- Estimated time: 3–4 hours

**Why now:** The current manual SSE implementation has no reconnection handling — if a user's connection drops mid-stream on a 30-second waterfall computation, they see a blank response with no recovery. Native SSE fixes this for free.

### 6. CIMD-Style Automated MCP Client Registration for Enterprise Onboarding — Impact: Med | Effort: Med

**Source:** [ServiceNow AI Gateway March 2026 release](https://www.servicenow.com/community/now-assist-articles/ai-gateway-what-s-new-in-the-march-2026-release/ta-p/3501670) — introduced CIMD (Client Identity Metadata Document) for automated MCP client registration. One registration covers all MCP servers on a host.

**Winston surface:** `backend/app/mcp/server.py` (server config), new `backend/app/mcp/cimd.py` (client identity endpoint), enterprise onboarding flow.

**Implementation sketch:**
- Currently, enterprise clients connecting to Winston's MCP server must manually configure auth tokens, tool lists, and server URLs — a multi-step IT setup
- Implement a CIMD-inspired registration endpoint: `GET /mcp/cimd.json` returns a Client Identity Metadata Document containing:
  ```json
  {
    "issuer": "https://api.winston.novendor.com",
    "registration_endpoint": "/mcp/register",
    "token_endpoint": "/mcp/token",
    "supported_tools": ["repe.*", "finance.*", "rag.*"],
    "auth_methods": ["bearer", "oauth2_client_credentials"],
    "scopes": ["read:fund", "read:asset", "read:lp", "write:scenario"]
  }
  ```
- Client (Claude Desktop, enterprise AI gateway) fetches CIMD → auto-configures connection → requests scoped token → starts using tools
- Combine with yesterday's MCP Server Card idea (`.well-known/mcp.json`) — the Server Card points to the CIMD endpoint for registration
- Estimated time: 2–3 days

**Enterprise value:** Reduces Winston onboarding from "2-hour IT call" to "paste one URL." This is the difference between a POC that stalls in IT review and one that goes live in a day. ServiceNow validated this pattern — Winston should adopt it immediately for the enterprise sales motion.

### 7. Qwen3.5 2B Multimodal GGUF for Local Document Thumbnails — Impact: Low | Effort: High

**Source:** [HuggingFace: Qwen3.5 2B GGUF models](https://hf.co/amkkk/Qwen3.5_2B_Abiliterate_All_Layers_Baked_GGUF_quantized) (March 10, 2026) — 2B parameter multimodal (image-text-to-text) model in GGUF format, suitable for local inference.

**Winston surface:** `backend/app/services/document_pipeline.py` (RAG ingestion), potential new `backend/app/services/local_inference.py`.

**Implementation sketch:**
- When documents are uploaded to Winston (PPMs, financials, offering memos), the RAG pipeline currently sends them to Claude for summarization — this is expensive for bulk uploads
- Deploy Qwen3.5 2B GGUF locally via `llama-cpp-python` for lightweight document tasks:
  - Generate 1-sentence summaries for document list views (currently blank or generic)
  - Extract key terms and entity tags (fund names, property addresses, dates) for search indexing
  - Pre-classify document types (PPM, subscription agreement, quarterly report, appraisal) before the user tags them
- The 2B model runs on CPU in <1s per page, no GPU required, cost = $0
- Gate behind a `WINSTON_LOCAL_INFERENCE` env var; fall back to Claude if disabled
- Estimated time: 3–5 days (including model download, inference setup, pipeline integration)

**Why consider this:** Bulk document ingestion is a known friction point — clients upload 50+ documents when onboarding a fund. Running 50 Claude API calls for summaries is slow and expensive. A local 2B model handles the lightweight preprocessing at zero marginal cost, reserving Claude for the complex cross-document analysis where it excels.
