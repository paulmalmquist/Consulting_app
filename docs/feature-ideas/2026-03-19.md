# Feature Ideas — 2026-03-19

## 🏆 Top Idea of the Day

**Extended Thinking `display: "omitted"` — Zero-Latency Thinking for Winston Chat**

Anthropic shipped a new `display` field for extended thinking this month (March 2026) that lets you omit thinking tokens from the streamed response while still preserving the cryptographic signature for multi-turn continuity. For Winston's REPE clients, complex queries like "run a waterfall sensitivity on IGF-VII across 8 cap rate scenarios" currently require Claude to think through the problem before text starts streaming — and that thinking content can be thousands of tokens of silent latency. With `display: "omitted"`, Winston's `ai_gateway.py` would pass `thinking={"type": "enabled", "budget_tokens": 8000, "display": "omitted"}` to the Claude API, causing text to start streaming immediately after the thinking signature is delivered. The first word of the answer would arrive seconds faster on every complex query — no new infrastructure, no cost increase, one parameter change. This is the highest-leverage API optimization available in Winston's stack today.

## 💡 All Ideas Today

### 1. Extended Thinking `display: "omitted"` for Faster Streaming — Impact: High | Effort: Low

**Source:** Anthropic Claude Developer Platform release notes, March 2026 — new `display` field for extended thinking that omits thinking blocks from the stream while preserving the signature.

**Winston surface:** `backend/app/services/ai_gateway.py` — specifically the Claude API call that enables extended thinking for complex REPE queries.

**Implementation sketch:**
- In `ai_gateway.py`, locate where `thinking={"type": "enabled", "budget_tokens": N}` is passed to the Anthropic SDK
- Add `"display": "omitted"` to that dict: `thinking={"type": "enabled", "budget_tokens": 8000, "display": "omitted"}`
- The `thinking_delta` SSE events will no longer stream to Winston's client — only a `signature_delta` will appear, then text begins immediately
- Store the `thinking` block (signature only) in session state for multi-turn continuity (required per Anthropic docs)
- Gate this behind a `WINSTON_THINKING_DISPLAY` env var initially so it can be toggled off if any issues surface
- Estimated time: 30–60 minutes including testing

**Why now:** Feature is brand-new (March 2026), the Python SDK already supports it transparently by forwarding the dict key. TypeScript SDK requires a type assertion. Winston uses Python on the backend — zero friction.

### 2. 1M Token Context Window for Full-Fund Document Mode — Impact: High | Effort: Low

**Source:** Anthropic March 2026 release — 1M token context window is now GA for Claude Opus 4.6 and Sonnet 4.6 at standard pricing, no beta header required.

**Winston surface:** RAG pipeline (`backend/app/services/ai_gateway.py`, document ingestion), new "Full Document Mode" toggle in the Winston chat workspace.

**Implementation sketch:**
- Remove the beta header previously required for >200k context requests (now unnecessary per release notes)
- Add a `full_document_mode: bool` flag to the AI gateway session config
- When enabled, bypass RAG chunking and instead inject all documents for the current environment/fund directly into context (PDFs, PPMs, subscription agreements, financials) up to 1M tokens
- Add a UI toggle in `ChatContextPanel.tsx` (right rail): "Document Mode: Chunked (fast) / Full (comprehensive)"
- Surface a context token count indicator showing how much of the 1M window is being used
- Estimated time: 2–4 hours including the context-stuffing logic

**Why this matters for REPE clients:** LPs and GPs asking cross-document questions like "Does the PPM's distribution policy match the waterfall we're running in the model?" currently require Winston to retrieve and re-rank chunks — with full context, it reads every document simultaneously. Juniper Square and Yardi do not offer this.

### 4. MCP Tasks Primitive for Long-Running REPE Jobs — Impact: Med | Effort: Med

**Source:** MCP 2026 Roadmap — Tasks primitive shipped as experimental, adding lifecycle semantics: retry on transient failure, progress events, expiry policies for result retention.

**Winston surface:** `backend/app/mcp/server.py`, long-running MCP tools (waterfall computation, scenario stress tests, capital account snapshot recompute).

**Implementation sketch:**
- Identify Winston tools that currently block waiting for results: `finance.run_waterfall`, `finance.stress_cap_rate` (100-scenario matrix), `finance.compare_scenarios`, snapshot recompute
- Wrap these in the MCP Tasks primitive: return a `task_id` immediately, then post progress events as computation proceeds
- Surface progress in the `ChatContextPanel` right rail: "Running 100-scenario stress test… 47% complete (47/100)"
- Add retry logic for transient DB connection failures (currently causes tool failures requiring manual re-prompt)
- On task completion, push a `tool_result` SSE event with the final data as normal
- This also fixes Bug 3 from META_PROMPT (Capital Account Snapshots requiring manual click) — the auto-trigger becomes a tracked Task
- Estimated time: 1–2 days

**Why this matters:** Currently if a user asks Winston to run a full waterfall sensitivity and it takes 8 seconds, the chat appears frozen. With Tasks, they see live progress in the right rail, dramatically improving perceived responsiveness.

### 5. PydanticAI Dependency Injection for MCP Tool Context — Impact: Med | Effort: Med

**Source:** PydanticAI (Pydantic team, 2025-2026) — brings FastAPI-style dependency injection to AI agents; tools declare their dependencies (db connection, user session, env context) as typed parameters auto-injected at runtime.

**Winston surface:** `backend/app/mcp/` — the MCP tool layer (83 tools currently).

**Implementation sketch:**
- Currently each MCP tool manually reads `env_id`, `fund_id`, `user_id` from the request context passed down from `ai_gateway.py`
- Adopt PydanticAI's `RunContext` / dependency injection pattern: define a `WinstonContext` dataclass with `env_id`, `fund_id`, `db_pool`, `user_id`, `auth_scope`
- Tools declare `ctx: RunContext[WinstonContext]` as their first argument — PydanticAI handles injection
- This reduces per-tool boilerplate by ~30–40 lines each and makes adding new tools significantly faster (junior dev can add a new MCP tool without understanding the full context-passing chain)
- Can be adopted incrementally — start with the 20 curated external-access tools from Priority 5
- Estimated time: 3–5 days for full rollout (1–2 days for pilot on 5 tools)

**Why now:** PydanticAI has matured significantly in early 2026 and is increasingly the standard pattern for FastAPI + AI integrations. Adopting it now before the 83-tool set grows further is the right time.

### 6. SGLang-Style Constrained Decoding for Structured REPE Outputs — Impact: Med | Effort: Med

**Source:** SGLang (2026) — fast LLM inference server with constrained decoding via JSON Schema or regex, eliminating malformed-output retries; also referenced in "Top 5 Structured Output Libraries for LLMs in 2026" alongside Instructor and PydanticAI.

**Winston surface:** `backend/app/services/ai_gateway.py` — specifically the fast-path for predictable analytical intents (fund metrics, deal comparisons, waterfall summaries).

**Implementation sketch:**
- For the fast-path analytical intents already defined in `repe_intent.py`, use Claude's `response_format` / structured output with tight Pydantic schema enforcement via Instructor
- Install Instructor (`pip install instructor`) and wrap the Anthropic client: `client = instructor.from_anthropic(Anthropic())`
- For structured result types (`FundMetricsCard`, `WaterfallResult`, `LPSummaryCard`), use `response_model=FundMetricsCard` — Instructor enforces the schema with automatic retry on malformed output
- This eliminates the current ad-hoc JSON parsing and retry logic in `ai_gateway.py` (~80 lines of error handling)
- Estimated time: 1–2 days

**Why now:** Instructor is production-stable in 2026, works transparently with the existing Anthropic Python SDK, and addresses a real pain point: occasional malformed structured outputs causing silent failures in Winston's analytical fast-path.

### 7. Breezy-Style "Unrealized Development Potential" Asset Score — Impact: High | Effort: High

**Source:** Breezy PropTech (2026) launched "Underbuilt" — a tool that uses zoning data + geospatial modeling + setback calculations to identify unrealized development potential per property, compressing weeks of analysis to seconds.

**Winston surface:** New MCP tool `repe.analyze_development_potential`, Deal Radar page, Asset Cockpit.

**Implementation sketch:**
- Integrate with Regrid (parcel + zoning data API) or a similar zoning data provider — they offer REST APIs with parcel-level zoning data for US properties
- New MCP tool `repe.analyze_development_potential(asset_id)`:
  1. Fetch parcel address from `re_asset` table
  2. Query Regrid API for zoning district, max FAR, height limits, setbacks
  3. Compute: current building SF vs. zoning-permitted SF → "unrealized entitlement" in sqft
  4. Return a structured score: `{ unrealized_sqft: 45000, current_use: "retail_2story", max_use: "mixed_use_6story", potential_value_range: "$12M–$18M", confidence: "medium" }`
- Surface in Asset Cockpit as a new "Development Potential" card (badge + score)
- Surface in Deal Radar as a filter: "Show assets with >20% unrealized entitlement"
- Estimated time: 3–5 days (gated on Regrid API access and pricing)

**Why this matters:** REPE firms tracking existing portfolios for value-add opportunities currently do this analysis manually with city zoning maps. An AI-native "unrealized potential" layer on every tracked asset is a genuine moat — no competitor offers this at the asset management level.
