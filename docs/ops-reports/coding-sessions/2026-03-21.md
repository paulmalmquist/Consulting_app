## Task Selected
**Bug 0: Raw Tool Call Spam Exposed in AI Chat UI** (CRITICAL/OPEN)
Source: `docs/LATEST.md` Active Bugs table — top priority per dispatch rules.
Users were seeing raw MCP tool names (e.g. `repe.get_asset`), retry attempts, and validation errors rendered directly in the chat UI instead of clean narrated steps.

## Complexity Assessment
**Worth Opus? Borderline.** The RunNarrator backend class and ToolActivityBlock frontend component already existed — the bug was that raw `tool_call` SSE events bypassed the narrator and leaked into the UI through three separate code paths. Understanding the full SSE pipeline (backend emission → frontend parsing → UI rendering) and tracing where raw names leaked required reading across 6 files. Sonnet could likely have handled the edits once the diagnosis was clear, but the multi-file diagnosis across a 3,900-line gateway file justified Opus.

## What Changed in the System
Three user-visible surfaces now show narrator labels instead of raw tool names:

1. **Thinking status indicator**: Was "Looking up get asset..." → Now "Fetching assets..."
2. **Context panel tool list**: Was "repe.get_asset" → Now "Fetching assets"
3. **Error messages**: Was raw validation text → Now "Some data could not be retrieved"

The existing RunNarrator (`run_narrator.py`) and ToolActivityBlock (`ToolActivityBlock.tsx`) were already correctly built — the bug was that the `tool_call` SSE event path bypassed them. The fix adds narrator labels to the `tool_call` events so all three rendering paths (response_block, tool_call, status) use clean labels.
