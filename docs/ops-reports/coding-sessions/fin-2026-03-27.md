## Item Selected

**P0 Bug: "No database connection" in Market Intelligence environment**

Source: `docs/market-digests/health/2026-03-26-market-health.md`

The Market Intelligence environment at `/lab/env/.../markets` displayed a permanent "No database connection" banner on every tab. All 34 active market segments, 16 intel briefs, and 33 feature cards in the database were invisible to users. The environment was completely non-functional.

## Root Cause

The markets page (`repo-b/src/app/lab/env/[envId]/markets/page.tsx`) used `getSupabaseBrowserClient()` to query trading tables directly from the browser. This function (in `repo-b/src/lib/supabase-client.ts`) returns `null` when `NEXT_PUBLIC_SUPABASE_URL` or `NEXT_PUBLIC_SUPABASE_ANON_KEY` are missing or set to "placeholder" values.

In the deployed environment, these browser-side Supabase credentials are placeholders. Other working lab pages (credit, pipeline, PDS) avoid this by routing through server-side API layers (`bosFetch()`, `apiFetch()`) that use the pg pool with `PG_POOLER_URL` — which IS configured correctly.

## Fix Applied

1. **Created `/api/v1/trading` API route** (`repo-b/src/app/api/v1/trading/route.ts`)
   - Uses server-side pg pool via `getPool()` from `@/lib/server/db`
   - Queries all 8 trading tables in parallel with the same ordering/filtering as the original Supabase queries
   - Returns structured JSON response with all trading data
   - Proper error handling with 503 (no pool) and 500 (query error) responses

2. **Updated markets page to use API route**
   - Removed `getSupabaseBrowserClient` import and direct Supabase queries
   - Replaced with `fetch("/api/v1/trading")` call
   - Simplified error handling (removed per-table error checks)
   - Net: -39 lines removed, +18 lines added in page.tsx
