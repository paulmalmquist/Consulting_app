## Items Implemented

### BUG-1 (P0) — SQL Generation Error Handling — FIXED

**File:** `backend/app/services/ai_gateway.py` (lines 1342–1380)

**Problem:** `generate_sql()` call was wrapped in a bare `except Exception: sql = ""` — when `OPENAI_API_KEY` is not set in Railway, `generate_sql()` raises `RuntimeError("OPENAI_API_KEY not configured")`, which was silently caught, swallowed, and fell through as empty SQL. Result: "I couldn't generate a SQL query" in ~18.5s with 0 Tools / 0 Tokens consumed. No log output, no user feedback, no Railway error trail.

**Fix:** Split into two typed exception handlers:
1. `except RuntimeError as e` — logs `ai.gateway.generate_sql.config_error` and yields a user-visible error token (`"AI query unavailable: ..."`) so the user knows the config is broken rather than getting a mystery failure
2. `except Exception as e` — logs `ai.gateway.generate_sql.error` with `error_type` and `intent[:100]` so Railway logs contain a real stack and error class

**Expected impact:** If OPENAI_API_KEY is set in Railway, the existing `generate_sql()` commit (fa9372dc) should work and AI queries will return results. If the key is missing, the user now sees a clear config error instead of a silent timeout.

### BUG-2 (P1) — Distributions "Total Paid" KPI Shows $0 — FIXED

**File:** `repo-b/src/lib/server/reFinanceOperations.ts` (line 1523, refactored to ~1523–1532)

**Problem (two compounded bugs):**
1. `paidAmount` was set using `(event.status === "processed") ? allocatedAmount : 0`. The 10 seeded paid events have `status = 'paid'` (not `'processed'`), so `paidAmount` was always 0.
2. Even if status matched, `allocatedAmount` came from `fin_distribution_payout` rows — but the seeded events have 0 payout rows, so `allocatedAmount = 0` anyway.

**Fix:**
- Introduced `isPaid` boolean: `status === "processed" || status === "paid"` — handles both the live status from the normal workflow and the seeded 'paid' status
- Added defensive fallback: when `isPaid && allocatedAmount === 0`, fall back to `declaredAmount` so the KPI shows the declared cash movement rather than $0. This is the correct behavior when payout rows are absent — the net_distributable amount is the best available approximation of what was paid.
- Status label also updated to use `isPaid` consistently

**Expected impact:** Total Paid KPI on the Distributions page should now show the sum of `net_distributable` for the 10 paid events (~$145M range) instead of $0.
