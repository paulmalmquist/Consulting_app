## What Was Done

### Fix 1: Asset Units/SF Display Logic (P1 — Task 3)

**Problem:** Phoenix Gateway Medical Office showing "256,577 units" despite being a medical office (should show SF). Multiple API routes preferred `pa.gross_sf` (legacy migration 270 column) over `pa.square_feet` (newer migration 285 column).

**Root cause identified:** Two issues found:
1. API routes queried `pa.gross_sf` which is from the older migration; newer seed scripts write to `pa.square_feet`. The COALESCE chain didn't fall back correctly.
2. Frontend display logic was not property-type-aware — it always showed units first regardless of asset type, and had no data quality guard for obviously bad values.

**Changes:**

| File | Change |
|---|---|
| `repo-b/src/app/api/re/v2/assets/route.ts` (GET + POST) | Changed `COALESCE(pa.gross_sf, 0)` to `COALESCE(pa.square_feet, pa.gross_sf, 0)` — prefers the column where newer seeds write data |
| `repo-b/src/app/api/re/v2/assets/[assetId]/route.ts` | Changed `pa.gross_sf AS square_feet` to `COALESCE(pa.square_feet, pa.gross_sf) AS square_feet` |
| `repo-b/src/app/api/re/v2/assets/[assetId]/reports/route.ts` | Same COALESCE fix |
| `repo-b/src/app/api/re/v2/investments/[investmentId]/assets/[quarter]/route.ts` | Reordered COALESCE to prefer `pa.square_feet` over `pa.gross_sf` |
| `repo-b/src/app/app/repe/assets/page.tsx` | Property-type-aware display: office/industrial/retail/medical/lab/data-center/self-storage prefer SF; multifamily/residential prefer units. Added 100K unit cap to catch bad data. |

**Confidence:** `[HIGH]` — TypeScript compiles clean. SQL COALESCE is backwards-compatible (falls back to gross_sf if square_feet is NULL). Display logic follows industry standard: commercial assets measured in SF, residential in units.

### Fix 2: Investment Sub-Records Seed Migration (P1 — Task 4)

**Problem:** All fund detail pages show "No type / No market / No valuation / Pending" for every investment because `repe_property_asset`, `re_asset_quarter_state`, and `re_investment_quarter_state` records are missing for most Meridian assets.

**Changes:**

| File | Change |
|---|---|
| `repo-b/db/schema/425_meridian_investment_backfill.sql` | New idempotent migration that backfills 3 layers of data for all Meridian assets |

**What the migration does:**
1. **Part I — Property Asset Backfill:** Finds all assets in the Meridian business missing `property_type`, `market`, `city`, `state`, or `msa`. Assigns property type based on asset name heuristics (multifamily, office, industrial, etc.). Assigns city/state/MSA from a deterministic rotating set of top REPE metros. Uses ON CONFLICT DO UPDATE with COALESCE to preserve any existing good data.
2. **Part II — Asset Quarter State Backfill:** Creates 5 quarters (2025Q1-2026Q1) of financial snapshots for assets missing them. NOI, revenue, opex, occupancy, debt, and asset value computed from property-type-specific cap rates and base metrics.
3. **Part III — Investment Quarter State Backfill:** Creates deal-level rollups by aggregating from asset quarter states. Includes NAV, committed/invested capital, IRR, equity multiple, and distributions.

**Confidence:** `[HIGH]` — Uses `ON CONFLICT DO NOTHING`/`DO UPDATE` for idempotency. Matches existing migration patterns (290, 349, 355, 359). Unique indexes on both quarter state tables verified (270_re_institutional_model.sql). Property type heuristics are reasonable for demo data and use deterministic hash-based assignments for consistency across re-runs.
