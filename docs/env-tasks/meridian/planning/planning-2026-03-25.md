### BUG-1 (P0) — SQL Generation Still Broken in Production (AI Analytics Lane)
**Health check status:** FAIL — "I couldn't generate a SQL query from your request" in ~18.5s, 0 Tools / 0 Tokens consumed.
**Root cause analysis from code review:**
- `ai_gateway.py` line 1343–1353: `generate_sql()` is imported at call time (`from app.sql_agent.combined_agent import generate_sql`). The function exists and is properly defined in `combined_agent.py` (line 145). The function itself uses `OPENAI_API_KEY` and `openai.AsyncOpenAI`. The most likely failure modes in production:
  1. `OPENAI_API_KEY` not set in Railway environment → `RuntimeError("OPENAI_API_KEY not configured")` is raised, caught by `except Exception: sql = ""`, and silently falls through. This would explain 0 Tokens consumed.
  2. Railway has not picked up commit `fa9372dc` yet (function was missing before that commit).
  3. A second import error downstream in `combined_agent.py` (e.g., `app.config` import failure).
- The `except Exception: sql = ""` catch in `ai_gateway.py` line 1349–1351 swallows all errors silently — this is why there's no error log.
**Fix:** Add explicit logging around the `generate_sql` call to surface the actual exception. Check Railway environment for `OPENAI_API_KEY`. Consider breaking the bare `except Exception` into two: one for `RuntimeError` (config problem, log and return error message) and one for unexpected errors.

```python
# ai_gateway.py ~line 1343
try:
    from app.sql_agent.combined_agent import generate_sql
    from app.sql_agent.catalog import catalog_text_dynamic
    catalog = catalog_text_dynamic(business_id=scenario.business_id)
    generated = await generate_sql(
        message=intent.original_message,
        catalog=catalog,
        business_id=scenario.business_id,
    )
    sql = generated.get("sql", "")
except RuntimeError as e:
    # Config/key error — surface to user
    yield _sse("token", {"text": f"AI query unavailable: {e}"})
    sql = ""
except Exception as e:
    emit_log(level="error", service="backend", action="ai.gateway.generate_sql.error",
             message=str(e), context={"intent": intent.original_message[:100]})
    sql = ""
```

**Files:** `backend/app/services/ai_gateway.py` (lines 1340–1360)
**Priority:** P0 — blocks AI analytics lane for all Meridian demos.

### BUG-2 (P1) — Distributions "Total Paid" KPI Shows $0 Despite 10 Paid Events
**Health check status:** Table data correct (10 Paid events display), KPI header wrong ($0).
**Root cause analysis from code review:**
- `reFinanceOperations.ts` line 1525: `const paidAmount = (event.status as string) === "processed" ? allocatedAmount : 0;`
- `totalPaid` sums `paid_amount` from all rows: `rows.reduce((sum, row) => sum + toNumber(row.paid_amount), 0)`
- The 10 seeded events have `status = 'paid'` in `fin_distribution_event` — but the code checks for `status === "processed"`, not `"paid"`.
- Additionally, `allocatedAmount` is computed from `fin_distribution_payout` rows. The health check confirmed "0 payout rows" on each event → `allocatedAmount = 0` even if the status matched.
- **Two separate bugs compounded:** (a) status string mismatch (`"paid"` vs `"processed"`), and (b) no `fin_distribution_payout` rows exist for the seeded events.
**Fix options (choose one):**
  1. **Preferred — seed payout rows:** Write a seeder that creates `fin_distribution_payout` records for the 10 paid events, split by payout_type (return_of_capital / preferred_return). This also fixes the investor breakdown drilldown.
  2. **Quick fix — status normalization:** Change the paid check in `reFinanceOperations.ts` to `(event.status === "processed" || event.status === "paid")` so the KPI reads from `declared_amount` when payout rows are absent. This is a band-aid.
  3. **Best fix — both:** Seed payout rows AND fix the status check to be defensive.

**Files:**
- `repo-b/src/lib/server/reFinanceOperations.ts` (line 1525 — status check)
- Seed: need to identify which service seeds `fin_distribution_event` records and add `fin_distribution_payout` inserts. The `fin_distribution_event` inserts appear to be done by a backend seeder triggered via the environment reset flow; check `backend/app/services/finance_repe.py` `create_distribution_event()` and add a companion payout seeder.
**Priority:** P1 — visible KPI data quality issue for every Meridian demo.
